## 2. 各関数の詳細設計

### setup() — 初期化処理

```
【処理の流れ】
1. ピンモードを設定する
   - PIN_POTENTIOMETER → 入力（アナログ）
   - PIN_BUZZER        → OUTPUT
   - PIN_BUTTON        → INPUT_PULLUP
2. Serial.begin(9600) でシリアル通信開始
3. 変数を初期化（currentState = 0, frequency = 0.0 など）
4. 起動確認としてブザーを短く鳴らす
```

---

### loop() — メインループ

```
【処理の流れ】

＜毎ループ実行すること＞
- ポテンショメータ値を読む（readPotentiometer）
- リセットボタン状態を読む
- millis()でタイミング管理

＜currentState が 0（待機中）のとき＞
- ポテンショメータ値を監視
- 左半分なら currentState = 1（周波数設定モード）
- 右半分なら currentState = 2（ランダム出力モード）

＜currentState が 1（周波数設定モード）のとき＞
- つまみ値から周波数を計算（convertKnobToFrequency）
- ブザーでその周波数を鳴らす（playTone）
- シリアルに周波数・音階名を表示（printToSerial）
- つまみが右半分になったら currentState = 2
- リセットボタンで currentState = 0

＜currentState が 2（ランダム出力モード）のとき＞
- ボタン押下が確定したときだけ、次の問題（音階）を出題する（手動方式）
- ボタンが押されていない間は同じ音を鳴らし続ける
- ボタンが押された瞬間に：
    1. selectRandomNote, calcFrequency, getNoteName で新しい音階を決定
    2. playTone で鳴らす
    3. printToSerial で表示
- つまみが左半分になったら currentState = 1
- リセットボタンで currentState = 0

【ランダム出力モードのボタン処理】
  - ボタン押下が確定した瞬間だけ nextRandomRequested=true にする
  - loop()内で nextRandomRequested==true のときだけ新しい問題を出題し、出題後にfalseに戻す

＜currentState が 3（エラー）のとき＞
- ブザーOFF
- シリアルにエラー内容表示
- リセットボタンで currentState = 0
```

## 3. 重要ロジックの詳細設計

### 3-1. チャタリング防止（デバウンス処理）

> ※ ボタンを使う場合は必ず設計してください。

```
【考え方】
  PIN_BUTTON は INPUT_PULLUP で使用するため、未押下=HIGH、押下=LOW で判定する。
  入力が変化した直後は接点が揺れるため、50ms 以上同じ値が続いたときだけ「確定入力」として扱う。
  押下が確定した瞬間に1回だけ resetRequested=true を立て、loop() 側で currentState=0 に戻す。

【処理の流れ】
  1. raw = digitalRead(PIN_BUTTON) を読む
  2. raw が lastButtonReading と異なる場合、lastDebounceTime = now に更新する
  3. now - lastDebounceTime >= DEBOUNCE_DELAY のときのみ入力を確定する
  4. 確定値が debouncedButtonState と異なる場合、debouncedButtonState を更新する
  5. debouncedButtonState が LOW になった瞬間だけ resetRequested=true にする
  6. loop() の各状態処理の最後で resetRequested を確認し、true なら currentState=0 に遷移
  7. 遷移後に resetRequested=false に戻す

【必要な変数（Section 1 に追加済みか確認）】
  lastDebounceTime   : unsigned long = 0   // 入力変化を検出した時刻
  DEBOUNCE_DELAY     : const int = 50      // チャタリング判定時間（ms）
  lastButtonReading  : bool = HIGH         // 前回の生入力
  debouncedButtonState : bool = HIGH       // 確定後のボタン状態
  resetRequested     : bool = false        // 押下確定イベント（1ショット）

【判定タイミング】
  - ボタン読取り周期: 10ms〜20ms（loop() 毎）
  - 押下確定条件: LOW が 50ms 連続
  - 解放確定条件: HIGH が 50ms 連続
```

---

### 3-2. millis() を使ったタイマー管理

```
【考え方】
  「前回実行した時刻」を記録しておき、「今の時刻 − 前回時刻 ≥ 周期」なら実行する。

【処理の流れ（例: LED点滅）】
  1. now = millis()
  2. now - lastMillis_LED >= LED_INTERVAL かどうか確認
  3. 条件を満たした場合: LEDのON/OFFを切り替え、lastMillis_LED = now
  4. 条件を満たさない場合: 何もしない（次のループで再チェック）

【自分のシステムで millis() を使う処理】
  1. now = millis() を毎ループ取得する
  2. now - lastUpdateTime >= 100ms のとき、ポテンショメータ読取りとエラーチェックを実行する
  3. 実行後は lastUpdateTime = now に更新する
  4. ランダム出力モードでは delay() を使わず、
     nextRandomRequested == true（押下確定イベント）になった瞬間だけ次の問題を出題する
  5. ブザー出力は tone(PIN_BUZZER, freq, duration) を使い、
     他の入力監視を止めない（非ブロッキング）
```

## 8. バグになりやすい箇所と解決策

### 1. ボタン処理のデバウンス
- **問題点:** デバウンス処理が正確に機能しない場合、ボタンの押下が複数回カウントされる可能性があります。
- **解決策:** `millis()`を使ったデバウンス処理を正確に実装し、押下確定のタイミングを明確にする。

### 2. ランダム出力モードの状態管理
- **問題点:** `nextRandomRequested`フラグが正しくリセットされない場合、次の問題が出題されない、または連続して出題される可能性があります。
- **解決策:** 問題出題後に必ず`nextRandomRequested = false`に戻す処理を確認する。

### 3. ポテンショメータの値範囲チェック
- **問題点:** ポテンショメータ値が範囲外（0未満や1023超）になった場合のエラー処理が抜けている可能性があります。
- **解決策:** 範囲外の値を検出した場合にエラー状態に遷移する処理を実装する。

### 4. 周波数計算の精度
- **問題点:** `convertKnobToFrequency()`での周波数計算が正確でない場合、音階がずれる可能性があります。
- **解決策:** 音階と周波数の対応表を確認し、計算式が正確であることをテストする。

### 5. 音階名の生成
- **問題点:** `getNoteName()`で音階名が正しく生成されない場合、シリアル出力が誤った情報を表示する可能性があります。
- **解決策:** 音階番号と音階名の対応が正しいか確認する。

### 6. エラー状態の処理
- **問題点:** エラー状態（`currentState = 3`）に遷移した後、リセットボタンで正しく待機状態に戻らない可能性があります。
- **解決策:** リセットボタンの処理が全ての状態で機能することを確認する。

### 7. タイミング管理
- **問題点:** `millis()`を使ったタイミング管理が正確でない場合、意図した周期で処理が実行されない可能性があります。
- **解決策:** タイミング管理のテストを行い、意図した周期で動作することを確認する。

### 8. グローバル変数の初期化
- **問題点:** グローバル変数が初期化されていない場合、予期しない動作を引き起こす可能性があります。
- **解決策:** `setup()`で全てのグローバル変数を明示的に初期化する。

### 9. シリアル出力の過剰
- **問題点:** シリアル出力が多すぎると、デバッグ時に情報が見づらくなる可能性があります。
- **解決策:** 必要最低限の情報のみを出力し、デバッグ用の出力は実装後に削除する。

### 10. 型の問題
- **問題点:** `float`型の周波数計算で丸め誤差が発生する可能性があります。
- **解決策:** 必要に応じて整数型で計算し、最後に`float`に変換する。
