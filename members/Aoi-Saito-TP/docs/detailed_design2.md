### `loop()` — メインループ

> ※ loop() は「状態ごとに何をするか」だけ書く。細かい処理は各関数に任せる。

```
【処理の流れ】

＜毎ループ実行すること＞
  - 入力を読む（readButton(), readSensor() などを呼ぶ）
  - 現在時刻を取得: now = millis()

＜currentState が 0（待機中）のとき＞
  - センサー値を監視する
  - 検知条件を満たしたら → currentState = 1

＜currentState が 1（動作中）のとき＞
  - メイン処理を行う
  - 終了条件を満たしたら → currentState = 2

＜currentState が 2（完了）のとき＞
  - 完了表示をする
  - リセットボタンが押されたら → currentState = 0

＜currentState が 3（エラー）のとき＞
  - エラー表示をする / リセットを待つ
```

**↓ 自分の loop() を設計してください**
```
【処理の流れ】

＜毎ループ実行すること＞
- 現在時刻 now = millis() を取得する
- 必要に応じて距離測定関数を呼ぶ
- 必要に応じてシリアル表示関数を呼ぶ
- currentState に応じてブザー制御を切り替える

＜currentState が 0（待機中）のとき＞
- 距離を監視する
- 距離が THRESHOLD_CM 以下なら currentState = 1 にする
- 遠い場合はブザーを止める

＜currentState が 1（警告中）のとき＞
- 距離に応じて音の高さを変えて鳴らす
- 距離が THRESHOLD_CM より大きくなったら currentState = 0 に戻す

＜共通処理＞
- 100msごとにシリアルへ距離を表示する

```

readSensor() — 距離を測定する

basic_design.md 2-2 との対応： readSensor()（超音波センサーの距離を取得して返す）


引数： なし
戻り値： long（距離 cm）


【処理の流れ】
1. TRIGピンを一度 LOW にする
2. 少し待つ
3. TRIGピンを HIGH にして超音波を送信する
4. すぐに TRIGピンを LOW に戻す
5. ECHOピンの HIGH 継続時間を測る
6. その時間から距離を cm に換算する
7. 測定結果を distance に入れる
8. 返り値として distance を返す

【エラー・異常ケース】


測定時間が取得できない場合:
0cm または異常値として扱う
距離が仕様外（例: 400cm超）の場合:
異常値として扱う


showDistance() — 距離を表示する

basic_design.md 2-2 との対応： showDistance()（シリアルモニタに距離を表示する）


引数： long distance（表示する距離）
戻り値： なし（void）


【処理の流れ】
1. 「Distance = 」のような文字列を表示する
2. 距離(cm)を表示する
3. 単位として「cm」を表示する
4. 改行する

【エラー・異常ケース】

「測定失敗」などの表示にしてもよい
異常値の場合:
そのまま表示せず、エラー表示に切り替えてもよい


calcFrequency() — 距離から周波数へ変換する

basic_design.md 2-2 との対応： calcFrequency()（距離を周波数に変換する）


引数： long distance（距離 cm）
戻り値： int（周波数 Hz）


【処理の流れ】
1. 距離が THRESHOLD_CM より大きい場合は 0 を返す
2. 距離が近いほど高い周波数になるように変換する
3. 周波数が MIN_FREQ 未満にならないようにする
4. 周波数が MAX_FREQ を超えないようにする
5. 計算した周波数を返す

【エラー・異常ケース】


距離が 0 または異常値の場合:
0 を返す
計算結果が範囲外の場合:
MIN_FREQ〜MAX_FREQ に収める


playToneByDistance() — 距離に応じてブザーを鳴らす

basic_design.md 2-2 との対応： playToneByDistance()（距離に応じて周波数を変えて鳴らす）


引数： long distance（距離 cm）
戻り値： なし（void）


【処理の流れ】
1. 距離を確認する
2. 距離が THRESHOLD_CM より大きければ stopBuzzer() を呼ぶ
3. 距離が範囲内なら calcFrequency() で周波数を求める
4. その周波数でブザーを鳴らす
5. currentState を 1 にする

【エラー・異常ケース】


周波数が 0 の場合:
ブザーを鳴らさない
距離が異常値の場合:
stopBuzzer() を呼ぶ

## 3. 重要ロジックの詳細設計

### 3-1. チャタリング防止（デバウンス処理）

> このシステムではボタン入力を使用しないため、チャタリング対策は不要。
```
【考え方】
  ボタンが押されたとき、50ms 以内の連続入力は「同じ1回の押下」として無視する。

【処理の流れ】
  1. ボタンのデジタル値を読む（digitalRead）
  2. 前回確定した時刻（lastDebounceTime）からの経過時間を計算する
  3. 経過時間 < DEBOUNCE_DELAY（例: 50ms）→ 無視する
  4. 経過時間 ≥ DEBOUNCE_DELAY → ボタンの状態として確定する
  5. lastDebounceTime を更新する

【必要な変数（Section 1 に追加済みか確認）】
  lastDebounceTime : unsigned long   // 前回確定した時刻
  DEBOUNCE_DELAY   : const int = 50  // チャタリング判定時間（ms）
```

---

### 3-2. millis() を使ったタイマー管理

```
【考え方】
  「前回実行した時刻」を記録しておき、
  「今の時刻 − 前回時刻 ≥ 周期」ならその処理を実行する。

【処理の流れ（例: 距離測定・表示更新）】
  1. now = millis() を取得する
  2. now - lastMillis_Sensor >= SENSOR_INTERVAL かどうか確認する
  3. 条件を満たした場合:
     - readSensor() を実行する
     - 必要なら showDistance() を実行する
     - lastMillis_Sensor = now に更新する
  4. 条件を満たさない場合:
     - 何もしない
  5. 次のループで再チェックする

【自分のシステムで millis() を使う処理】
  - 距離測定：100msごと
  - シリアル表示：100msごと
  - ブザー制御：距離測定と同時
```

### 3-3. その他の重要ロジック（任意）

距離に応じた周波数変換

【処理の流れ】
1. 距離が THRESHOLD_CM より大きいか確認する
2. 大きい場合は 0Hz としてブザー停止扱いにする
3. 範囲内の場合は、距離が近いほど高い周波数になるよう変換する
4. 周波数が MIN_FREQ 未満なら MIN_FREQ に補正する
5. 周波数が MAX_FREQ を超える場合は MAX_FREQ に補正する

【入力値と出力値の関係】
- 入力: 距離(cm)
- 出力: 周波数(Hz)
- 距離が小さいほど出力周波数が大きくなる
- 距離が閾値を超えると出力は 0

## 6. AIレビュー記録

### Q1: 実装上の問題確認

> 「この詳細設計書に書いた関数と処理フローをもとに Arduino でコードを書きます。バグになりやすい箇所・処理の抜け・型の問題はありますか？」

**AIの回答（要約）：**

millis() の比較は、unsigned long 同士で扱わないとオーバーフロー時に不具合が出やすい。
readSensor() では、距離算出時に除算や変換式の精度に注意が必要です。long と int の混在で桁落ちしないようにした方がよい。
calcFrequency() は、距離が 0 や異常値のときにそのまま計算すると不自然な値になります。入力チェックを先に入れる必要がある。
playToneByDistance() と stopBuzzer() の両方でブザー制御が重複しないように整理すると安全。
delay() を通常処理に使うと、距離変化への反応が遅れるため、初期化以外では使わない方針でよい。
**対応した内容：**
タイマー変数を unsigned long に統一した
距離の異常値（0、範囲外）を先に除外する設計にした
calcFrequency() の前に距離チェックを入れる流れにした
ブザー停止処理を stopBuzzer() にまとめた
通常動作は millis() ベースにして delay() を避ける方針にした

---

### Q2: 単体テスト仕様の確認

> 「Section 5 の単体テスト仕様書で、各関数の動作が正しく検証できていますか？テストが不足している項目や、境界値テストが必要な箇所を教えてください。」

**AIの回答（要約）：**
基本的な入出力テストはできていますが、境界値テストをもう少し追加すると安心。
たとえば calcFrequency() では、THRESHOLD_CM ちょうどの値、THRESHOLD_CM + 1、0cm の確認が必要。
readSensor() では、正常範囲だけでなく、最大測定距離付近や反射不足時の挙動も確認した方がよい。
millis() を使う処理では、短時間しか待たなくても周期更新されるかを確認するとよい。
ブザー制御は、鳴動開始直後と停止直後の切り替わりもテスト対象に入れると完成度が上がる。
**対応した内容：**
THRESHOLD_CM ちょうどの境界値テストを追加する方針にした
0cm や範囲外の値を異常値として確認する項目を入れた
センサーの最大付近・測定失敗時の確認を追加した
millis() の周期更新確認をテスト項目に含めた
ブザーの開始・停止切り替え確認を追加した
