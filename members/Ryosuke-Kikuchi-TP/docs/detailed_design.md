### `setup()` — 初期化処理

```
【処理の流れ】
1. ピンモードを設定する
   - PIN_BUTTON  → INPUT_PULLUP
   - PIN_LED_*   → OUTPUT
   - PIN_BUZZER  → OUTPUT

2. ライブラリの初期化（使うものだけ）
   - 例: lcd.begin(16, 2)
   - 例: servo.attach(PIN_SERVO)

3. Serial.begin(9600)（デバッグ用）

4. 起動確認（任意）: 緑LEDを1秒点灯して消灯
```

**↓ 自分の setup() を設計してください**
```
【処理の流れ】
1. ピンモードを設定する（PIN_LED_BLUE/PIN_LED_RED: OUTPUT, PIN_BUZZER: OUTPUT, キーパッド行列ピン初期化）
2. 必要なライブラリの初期化（例: 7セグ表示用など、任意）
3. Serial.begin(9600)（デバッグ用）
4. 起動確認として青LEDを1秒点灯して消灯
```

### `loop()` — メインループ

> ※ loop() は「状態ごとに何をするか」だけ書く。細かい処理は各関数に任せる。

```
【処理の流れ】

＜毎ループ実行すること＞
  - 入力を読む（readButton(), readSensor() などを呼ぶ）
  - 現在時刻を取得: now = millis()

＜currentState が 0（待機中）のとき＞
  - センサー値を監視する
  - 検知条件を満たしたら → currentState = 1

＜currentState が 1（動作中）のとき＞
  - メイン処理を行う
  - 終了条件を満たしたら → currentState = 2

＜currentState が 2（完了）のとき＞
  - 完了表示をする
  - リセットボタンが押されたら → currentState = 0

＜currentState が 3（エラー）のとき＞
  - エラー表示をする / リセットを待つ
```

**↓ 自分の loop() を設計してください**
```
【処理の流れ】

＜毎ループ実行すること＞
  - キーパッド入力を読む（readKeypad()）
  - 現在時刻を取得: now = millis()

＜currentState が 0（待機中）のとき＞
  - 時間設定・スタートキー受付（setTimeFromKeypad()）
  - 設定完了＆スタートで currentState = 1, remainSeconds = setSeconds

＜currentState が 1（計測中）のとき＞
  - 1秒ごとにremainSecondsを減算
  - showTimeSerial()で残り時間表示
  - remainSeconds==0で currentState = 2

＜currentState が 2（終了通知）のとき＞
  - updateNotify()でLED点滅・ブザー制御
  - 停止キーまたはリセットキー押下時のみ currentState = 0
```

### `readKeypad()` — キーパッド入力を取得する

**basic_design.md 2-2 との対応：** キーパッド入力を取得し操作キーを判定

**引数：** なし

**戻り値：** char（未入力時は '\0'）

```
【処理の流れ】
1. now = millis() を取得し、前回確定時刻から50ms未満なら入力を無効化する
2. 行列スキャンで押下キーを検出し、対応する文字（0-9, 開始, 停止, リセット）へ変換する
3. 有効キーなら lastDebounceTime を更新し、keyInput として返す

【エラー・異常ケース】
- 未定義キーが来た場合: '\0'を返して状態を変更しない
```

---

### `setTimeFromKeypad(char key)` — キー入力から設定時間を確定する

**basic_design.md 2-2 との対応：** キー入力から設定時間を確定する

**引数：** key（char）: 入力キー

**戻り値：** bool（設定確定で true）

```
【処理の流れ】
1. 数字キーなら setSeconds の入力バッファへ追加する
2. 開始キーなら setSeconds の範囲チェック（1〜5999秒）を行う
3. 範囲内なら true を返し、範囲外なら false を返して直前の有効値を維持する

【エラー・異常ケース】
- 0秒または上限超過の場合: 受け付けない
```

**引数：** mm, ss（uint8_t）: 分・秒

**戻り値：** void

```
【処理の流れ】
1. remainSeconds から mm, ss を計算する
2. "MM:SS" 形式で Serial に出力する
3. 状態（待機/計測/通知）もあわせて出力する

【エラー・異常ケース】

```

### `updateLedByState(uint8_t state)` — 状態に応じてLEDを制御する

**basic_design.md 2-2 との対応：** 状態に応じて青/赤LEDを制御

**引数：** state（uint8_t）: 現在状態

**戻り値：** void

```
【処理の流れ】
1. 待機中: 青OFF, 赤OFF
2. 計測中: 青ON, 赤OFF
3. 終了通知: 青OFF, 赤LEDを1秒に5回で点滅

【エラー・異常ケース】

```

### `notifyByBuzzer(unsigned long durationMs)` — ブザーで終了通知を行う

**basic_design.md 2-2 との対応：** 終了時にブザーを鳴らす

**引数：** durationMs（unsigned long）: 鳴動時間ms

**戻り値：** void

```
【処理の流れ】
1. 通知開始から durationMs の間だけブザーを鳴らす
2. 経過後はブザーを停止する
3. 停止キー押下時は即時停止する

【エラー・異常ケース】
- durationMs が0の場合: 鳴動せず即終了
```

---

### `updateNotify()` — 終了通知中のLED点滅とブザー制御

**basic_design.md 2-2 との対応：** 終了通知中のLED点滅とブザー制御

**引数：** なし

**戻り値：** void

```
【処理の流れ】
1. notifyStartMillis からの経過時間を計算する
2. 終了通知中は、赤LED点滅（1秒に5回）とブザー鳴動を維持する
3. 停止/リセット入力があったときのみ通知終了し、待機中へ戻す

【エラー・異常ケース】
- タイマー値が不整合の場合: 通知を停止して待機中へ戻す
```

## 3. 重要ロジックの詳細設計

### 3-1. チャタリング防止（デバウンス処理）

> ※ ボタンを使う場合は必ず設計してください。

```
【考え方】
  ボタンが押されたとき、50ms 以内の連続入力は「同じ1回の押下」として無視する。

【処理の流れ】
  1. ボタンのデジタル値を読む（digitalRead）
  2. 前回確定した時刻（lastDebounceTime）からの経過時間を計算する
  3. 経過時間 < DEBOUNCE_DELAY（例: 50ms）→ 無視する
  4. 経過時間 ≥ DEBOUNCE_DELAY → ボタンの状態として確定する
  5. lastDebounceTime を更新する

【必要な変数（Section 1 に追加済みか確認）】
  lastDebounceTime : unsigned long   // 前回確定した時刻
  DEBOUNCE_DELAY   : const int = 50  // チャタリング判定時間（ms）
```

---

### 3-2. millis() を使ったタイマー管理

```
【考え方】
  「前回実行した時刻」を記録しておき、「今の時刻 − 前回時刻 ≥ 周期」なら実行する。

【処理の流れ（例: LED点滅）】
  1. now = millis()
  2. now - lastMillis_LED >= LED_INTERVAL かどうか確認
  3. 条件を満たした場合: LEDのON/OFFを切り替え、lastMillis_LED = now
  4. 条件を満たさない場合: 何もしない（次のループで再チェック）

【自分のシステムで millis() を使う処理】
  - キーパッド入力スキャン: 10〜20ms 周期（チャタリング対策と応答性を両立）
  - タイマー減算: 1000ms 周期（1秒単位で減算）
  - 終了通知制御: 100〜200ms 周期（赤LED点滅とブザー制御）
  - （任意）7セグ桁更新: 2〜5ms/桁（ちらつき防止）
```

---

### 3-3. その他の重要ロジック（任意）

> **【任意】** 複雑なロジックがある場合のみ記入してください。
> 例：「距離に応じたLED点灯パターン」「ゲームの衝突判定」「温度の閾値判定」

```
【処理の流れ】
1. setSeconds が確定したら remainSeconds = setSeconds として計測を開始する
2. 計測中は 1000ms ごとに remainSeconds を 1 減算する
3. remainSeconds が 0 になった時点で終了通知状態へ遷移する
4. 終了通知状態は、停止キーまたはリセットキー押下時のみ待機中へ戻る

【入力値と出力値の関係】
- 入力: setSeconds（1〜5999秒）
- 出力: remainSeconds（毎秒減算）、state遷移（待機中→計測中→終了通知。終了通知から待機中への復帰は停止/リセット入力時のみ）
```

## 6. AIレビュー記録

> グループレビューの前に必ず実施してください。

### Q1: 実装上の問題確認

> 「この詳細設計書に書いた関数と処理フローをもとに Arduino でコードを書きます。バグになりやすい箇所・処理の抜け・型の問題はありますか？」

**AIの回答（要約）：**

- 状態は3つで閉じており、待機へ戻る経路も確保されている。
- `remainSeconds` と `setSeconds` を `uint16_t` にしているため型の選択は妥当。
- 入力デバウンスを `readKeypad()` に集約するとバグ混入を減らせる。
- 通知中断（停止/リセット）を `updateNotify()` 内で常時監視する実装が必要。

**対応した内容：**

- 関数ブロックを `readKeypad`/`setTimeFromKeypad`/`updateNotify` など実装単位で分離。
- 異常ケース（範囲外入力、未定義キー、未定義state）を各関数に追記。
- タイミング処理を `millis()` 周期で明示し、`delay()`依存を避ける方針を固定。

---

### Q2: 単体テスト仕様の確認

> 「Section 5 の単体テスト仕様書で、各関数の動作が正しく検証できていますか？テストが不足している項目や、境界値テストが必要な箇所を教えてください。」

**AIの回答（要約）：**

- 入力系は「正常入力・チャタリング・境界値（0秒/上限）」がそろっており妥当。
- 出力系は状態別LED制御とブザー時間の検証ができている。
- 追加で「通知中に停止したときの即時反映」を並行動作テストに含めるべき。

**対応した内容：**

- 5-3 に通知中停止のテストを追加。
- 5-1 に時間設定の境界値テスト（0秒）を明記。
- デバッグ出力計画に `remainSeconds` 監視を追加。
