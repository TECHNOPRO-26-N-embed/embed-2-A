## 1-2. 状態遷移（ガジェットの「ふるまい」設計）

> ※ ガジェットが「どんな状態を持つか」を定義します。
> この設計が loop() の骨格になります。

**記入例：**
```
[電源ON / 初期化]
  ↓（初期化完了）　　　　　　　　　　　　　
[待機中] ──（温度>=35.0℃を検知）──→ [計測中] ──（30秒経過 かつ 値が安定）──→ [判定・通知]
   ↑               │                           │                                   │
   │               └──（温度<35.0℃が継続）──→ [測定エラー] ──（再測定条件成立）──┘
   └──────────────────────────────（通知完了 / タイムアウト）──────────────────────┘
```

**各状態の意味：**

| 状態名 | この状態のとき、何をしているか | LEDやブザーなど出力の様子 |
|:--|:--|:--|
| 待機中 | 100ms周期で温度センサー値を監視し、測定開始条件を判定する | LED消灯、ブザー停止、LCDは「Ready」または「----」 |
| 計測中 | 測定開始時刻を基準に温度推移を取得し、表示値を更新する | 進捗を音階（ドレミファソラシド）で通知、LEDは温度段階で下から積み上げ点灯、LCDに現在温度 |
| 判定・通知 | 30秒時点の最終温度帯を判定し、結果に対応した音とLEDパターンを確定出力する | 39.0℃超なら全段点灯（赤まで）後に点滅、その他は最終段を保持 |
| 測定エラー | 35℃未満が継続して有効測定とみなせない状態を通知する | 力が抜ける音、LED消灯、LCDにエラー表示 |

> [!TIP]
> すべての状態から「抜け出せるルート」があるか確認してください。
> 「このボタンを押せば必ずリセットに戻る」が保証されていますか？

### 2-3. タイミング設計（delay vs millis）

> ※ `delay(N)` はN ms間、**すべての処理**が止まります。
> 複数の処理を並行させたい場合は `millis()` を使います。

| 処理 | 必要な周期 | delay / millis | 判断の理由 |
|:--|:--|:--|:--|
| 温度センサー読み取り | 100ms | **millis** | 計測中も他出力を止めずに連続監視するため |
| 音階更新（進捗通知） | 200ms | **millis** | 30秒進行を滑らかに音で知らせつつ表示処理と並行するため |
| LED段階表示更新 | 100ms | **millis** | 状態遷移に即応して色変更するため |
| 4digit表示更新 | 50ms | **millis** | 桁更新を安定させ、ちらつきを抑えるため |
| 起動時の確認音 | 150ms | **delay** | 初期化直後の単発処理で並行性が不要なため |

> [!CAUTION]
> `delay()` を使う場合、その間はボタン入力もセンサー読み取りも**すべて止まります**。

## 4. 異常系・エラー処理の方針

| 異常ケース | 発生条件 | 検知方法 | 対応方針 |
|:--|:--|:--|:--|
| 測定未成立 | 35℃未満の値が一定時間継続 | tempC < 35.0 が5秒以上継続 | 測定エラー状態へ遷移し、エラー音+LED消灯 |
| センサー異常値 | ADC値が極端に低い/高い | tempC < 20.0 または tempC > 45.0 | 値を無効化し、直前の有効値を保持 |
| アナログノイズ | 閾値付近で温度が小刻みに変動 | 直近5サンプルの差分が過大 | 移動平均で平滑化し、状態遷移にヒステリシスを導入 |
| 判定タイムアウト | 計測開始後30秒経っても安定しない | millis()差分が30,000ms超 | 現時点の平均値で仮判定し、再測定を促す |
| 高温緊急通知 | 30秒時点で39.0℃を超える | finalTempC > 39.0 を判定・通知遷移時に検出 | 青〜赤を全点灯した後、全段を点滅して緊急通知 |
| 4digit表示不整合 | 表示更新が途切れる | 更新周期超過を監視 | 1秒以上更新なしで `Err` を表示 |

## 7. AIレビュー記録

> グループレビューの前に必ず実施してください。

### Q1: 設計の妥当性確認

> 「この基本設計書をもとに Arduino UNO R3 でコードを書きます。ピン競合・ライブラリ競合・状態遷移の抜け・delay() の問題点はありますか？」

**AIの回答（要約）：**
- D13（ブザー）、D4-D12（LED）、D2-D3（4digit）、A0（温度）でピン競合はありません。
- 状態遷移は待機中/計測中/判定・通知/測定エラーで閉路が成立しており、停止状態がありません。
- 周期処理は millis() 主体で、計測中の delay() 依存がないため応答性を維持できます。

**対応した内容：**
- 3-1にピン割り当てを固定し、競合なしを明記。
- 1-2に「測定エラー -> 待機中」の復帰経路を追加。
- 2-3に delay() を起動時単発のみに限定。

---

### Q2: チャタリング・delay・ノイズの確認

> 「この設計にチャタリング対策は含まれていますか？delay() を使っている箇所で問題はありますか？アナログセンサーを使う場合、ノイズ対策は必要ですか？」

**AIの回答（要約）：**
- 本設計はボタン入力を持たないため、チャタリング対策は不要です。
- 温度閾値周辺でノイズによる誤判定が起きるため、移動平均とヒステリシスが有効です。
- 4digit更新停止時の異常表示を持つとデバッグ容易性が上がります。

**対応した内容：**
- 4章に移動平均・ヒステリシスの方針を追加。
- 4章に4digit表示不整合の検知と `Err` 表示を追加。

---

### Q3: 要件への対応漏れ確認（任意）

> **【任意】** トレーサビリティ表（Section 5）を書いた場合に実施してください。

**対応した内容：**
